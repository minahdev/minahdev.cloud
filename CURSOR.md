# Cursor 하네스 (이 저장소)



Cursor에서 돌아가는 에이전트·채팅의 행동을 **경계(harness)** 로 묶는다. 의도는 안드레아 카파시가 정리한 LLM 코딩 보조의 **네 원칙**(구현 전 사고 → 단순성 → 정밀한 수정 → 목표 중심 실행)과 같다. 원칙의 **전문·예시**는 `CLAUDE.md`를 따른다. 이 파일은 **Cursor에 붙이는 하네스**(규칙 쌓기, 도구 사용, 검증 루프)만 다룬다.



> **트레이드오프:** 속도보다 신중함·검증 가능성. 사소한 편집은 맥락에 따라 완화해도 된다.



---



## 규칙이 쌓이는 순서 (하네스 층)



| 층 | 역할 |

|----|------|

| `CLAUDE.md` | 카파시식 네 원칙의 **본문**(가정·단순성·수술적 diff·검증 가능 목표). |

| **루트 `.cursorrules`** | monorepo 전역 **짧은 끈** + `docs/` 필수 참조. |

| **`.cursorrules` (본 디렉터리)** | `backend/` 작업 시 **항상** 적용되는 최소 하네스. |

| `frontend/.cursorrules` | `frontend/` 작업 시 (해당 폴더 기준). |

| **`docs/README.md`** | **코딩 규칙 인덱스** — 구현 전 반드시 열람. |

| **`docs/DevOps/`** | 스택별 **상세 규칙**(React, FastAPI 등). |

| `CURSOR.md` (본 문서) | Cursor **워크플로·도구**와 네 원칙을 어떻게 맞출지 정리. |



충돌 시: **사용자가 명시한 요청** → **해당 `docs/` 규칙** → `CLAUDE.md` / 하네스.



---



## docs/ 필수 워크플로 (코딩 전)



에이전트는 **파일을 수정하기 전에** 다음을 수행한다.



1. **`docs/README.md`** 를 읽어 작업 영역에 맞는 문서를 확인한다.

2. 해당 문서를 **Read** 한 뒤 구현한다 (추측으로 규칙을 만들지 않는다).



| 작업 경로 | 반드시 읽을 문서 |

|-----------|------------------|

| `backend/apps/` (secom, main, DB 등) | `docs/DevOps/Backend/BACKEND_RULES.md` |

| `backend/apps/titanic/` | `docs/DevOps/Backend/BACKEND_RULES.md` + `docs/타이타닉개발/james_fastapi_init.md` |

| `frontend/` | `docs/DevOps/Frontend/REACT_RULES.md` |



Cursor 채팅에서 예: `@docs/README.md` `@docs/DevOps/Backend/BACKEND_RULES.md`



새 규칙 문서를 추가하면 **`docs/README.md` 표에 등록**한다.

## Cursor 도구와 네 원칙 매핑



### 1. 구현 전 사고



- 파일을 고치기 전에 **관련 코드와 `docs/` 규칙**을 읽고 경계를 잡는다. 추측으로 구조를 상상하지 않는다.

- 요구가 애매하면 **질문**하거나, 성공 기준을 한 줄로 제안해 사용자가 고르게 한다.



### 2. 단순성 우선



- “나중에 쓸 수도” 있는 추상화·설정·기능을 넣지 않는다.

- 터미널·스크립트도 **요청을 만족하는 최소 명령**만 실행한다.



### 3. 정밀한 수정



- **요청과 무관한 파일·줄**을 건드리지 않는다. 포맷만 바꾸는 drive-by 정리를 하지 않는다.

- 편집 후 **변경한 파일** 위주로 린트·타입 진단을 확인하고, **이번 diff로 생긴** 문제만 고친다.



### 4. 목표 중심 실행



- 빌드·테스트·린트가 있으면 **가능한 한** 같은 루프로 끝을 맞춘다: 목표 정의 → 실행 → 실패 시 원인 범위만 수정.

- 테스트가 없는 작업이면 **수동 확인 절차**(예: “이 화면에서 이 버튼을 누르면 …”)를 성공 기준으로 적어 둔다.



---



## 권장 루프 (하네스의 기본 형태)



1. **`docs/README.md` + 영역 규칙** 확인.

2. **성공 기준**을 말로 고정한다(테스트 이름, 명령 한 줄, 또는 확인 절차).

3. **최소 변경**으로 구현한다.

4. **검증**한다(테스트·빌드·수동 체크 중 저장소에 맞는 것).

5. 통과할 때까지 **2–4만** 반복한다. 넓게 리팩터링하지 않는다.



---



## 에이전트에게 맡기지 말 것 (하네스 밖)



- 비밀·키·프로덕션 데이터를 규칙 파일에 넣지 않는다.

- “전부 다 고쳐줘”처럼 **검증 불가능한** 범위만 던지지 않는다. 범위를 쪼개거나 기준을 붙인다.



---



## 이 하네스가 잘 먹히는지



- diff가 **요청에만** 붙는지.

- **`docs/` 규칙을 읽은 뒤** 기존 패턴과 맞는지.

- 재질문·재작업 없이 **검증 루프**로 끝나는지.

- 불필요한 추상화·파일 확장이 줄었는지.



상세 체크리스트는 `CLAUDE.md`의 마지막 절을 병행한다.
